# README.md

## File Integrity Checker

A command-line tool for computing and verifying file integrity using cryptographic hash functions.

### Features

- **Secure Hashing**: Supports SHA-256 and SHA-512 algorithms
- **Large File Support**: Uses streaming to handle files of any size with constant memory usage (4KB)
- **Integrity Verification**: Compare computed hash against expected value
- **Low-level OpenSSL API**: Direct use of SHA256/SHA512 functions (not EVP)
- **Robust Error Handling**: Comprehensive error checking and reporting
- **User-friendly**: Clear command-line interface with helpful usage information

### Prerequisites

- GCC compiler
- OpenSSL development libraries (`libssl-dev` on Ubuntu/Debian)

### Build

```bash
make
```

Or manually:
```bash
gcc -Wall -Wextra -O2 -o filehash filehash.c -lssl -lcrypto
```

### Usage

```bash
./filehash [-a algorithm] filename [expected_hash]
```

#### Options

- `-a algorithm`: Specify hashing algorithm (`sha256` or `sha512`)
  - Default: `sha256`
- `-h, --help`: Display help message

#### Examples

```bash
# Compute SHA-256 hash (default)
./filehash myfile.txt

# Compute SHA-512 hash
./filehash -a sha512 myfile.txt

# Verify file integrity
./filehash myfile.txt e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855

# Show help
./filehash --help
```

### Supported Algorithms

- **sha256** (default)
  - Output: 256-bit hash (32 bytes, 64 hex characters)
  - Security: Strong, industry standard
  - Use case: General purpose file integrity
  
- **sha512**
  - Output: 512-bit hash (64 bytes, 128 hex characters)
  - Security: Very strong, enhanced security
  - Use case: High-security requirements

#### Design Decisions

**Streaming vs. One-shot Loading:**
- Uses streaming (SHA256_Init/Update/Final) instead of loading entire file into memory
- Advantage: Constant memory usage (4KB buffer) regardless of file size
- Supports files larger than available RAM

**Algorithm Selection:**
- SHA-256: Default algorithm, balances security and performance
- SHA-512: Enhanced security for sensitive applications
- Both are part of NIST-approved SHA-2 family

**Buffer Size:**
- 4KB chunks provide good balance between I/O efficiency and memory usage
- Standard choice for file processing applications

### Security Considerations

- **Hash Strength**: SHA-256 and SHA-512 are cryptographically secure
- **Collision Resistance**: Both algorithms resist collision attacks
- **Large File Support**: Streaming prevents memory exhaustion attacks
- **Input Validation**: All user inputs are validated

### Limitations

- Only supports SHA-256 and SHA-512 (by design for this assignment)
- Does not store or manage hash values; user must maintain expected hashes separately
- Binary files are processed as-is; no special handling for different file types

### Exit Codes

- `0`: Success
- `1`: Error (file not found, invalid algorithm, etc.)
